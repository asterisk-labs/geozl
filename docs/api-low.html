low-level api

One recipe, by hand.

A recipe string is a compression graph. This page builds one.

The high-level page ends with a winner, planar>zigzag>transpose>entropy. Here that same graph gets assembled node by node in OpenZL and returns the same 116.5 kB frame, because the string was never anything more than a shorthand for these four calls.

OpenZL is a graph over typed streams. A node rewrites a stream and hands you its outputs. A graph closes a branch by turning a stream into bytes. Compression walks the tree and the frame records the walk.

Which is why DCtx needs no compressor to read a frame. It replays what the frame says happened.

the model openzl 0.2.0 · format 24

Node · a codec

Rewrites one stream into one or more. It never decides where its outputs go, so a node on its own is not a graph. zigzag, transpose_split and planar are all nodes.

Graph · a destination

Takes a stream and is responsible for it from there. entropy and zstd are graphs that end in bytes. A node plus a destination for every output is also a graph.

Frame · the walk

Records which codecs ran and in what order, so DCtx replays it without ever seeing your Compressor. One decompressor, every frame.

Everything on this page runs against openzl 0.2.0 from PyPI, writing frame format version 24. Custom codecs are the one exception to the universal decompressor, and the codec section covers exactly what that costs you.

streams are typed and the type gates the codec

Every stream in the graph carries one of four types, and each codec declares which it accepts. Half of the errors you will hit at this level are type errors, not logic errors, so it pays to read the pipeline as a sequence of types before reading it as a sequence of transforms.

The gates are narrow on purpose. zstd only takes Serial, because it is a byte compressor and has no notion of an element. field_lz takes Struct or Numeric, because it matches on field boundaries. entropy takes all three, because an order-0 model only needs a symbol alphabet.

That is where convert_num_to_struct_le earns its place in the recipe. transpose_split wants Struct, zigzag produced Numeric, and the conversion is a retype with no bytes moved.

why the recipe reads the way it does The grammar of a recipe string hides the conversions. planar>zigzag>zstd is really four nodes, since zstd cannot see a Numeric stream and geozl inserts convert_num_to_serial_le for you. At this level you insert it yourself.
typeelementwhat accepts it
Serial1 byte, flatA plain byte string with no structure. zstd, entropy, flatpack, store. Every transpose lane lands here.
StructN bytes, fixedFixed-width records whose interior is opaque. transpose_split, field_lz, entropy, tokenize. This is the type that knows an element is 2 bytes wide.
Numeric1, 2, 4 or 8 bytesIntegers in native order, the only type with arithmetic meaning. zigzag, delta_int, bitpack, range_pack, and every geozl predictor.
StringvariableVariable-length fields carried with a separate lengths stream. Unused by geozl, which only ever sees rectangular numeric arrays.

A tile enters as zl.Input(zl.Type.Numeric, arr). The width comes from the array dtype, so an int16 and a uint16 both arrive as 2-byte elements, which is the reason decompress needs a dtype. The frame keeps the width and drops the sign.

build the recipe four nodes, inside out

Graphs are assembled from the leaf backwards. Calling a node gives you a graph that runs that node and forwards to the successor you named, so the last stage of the pipeline is the first thing you construct and the head node is the last.

entropy ← transpose_split ← convert ← zigzag ← planar
python · assemble the graph
import numpy as np
import openzl.ext as zl

compressor = zl.Compressor()

# the head is a codec OpenZL does not ship, so it registers first
planar = compressor.register_custom_encoder(PlanarEncoder())
planar = compressor.parameterize_node(
    planar, local_params=zl.LocalParams({WIDTH: 512})
)

# then the tail, leaf first
lanes  = zl.nodes.TransposeSplit()(compressor, successor=zl.graphs.Entropy())
packed = zl.nodes.ConvertNumToStructLE()(compressor, successor=lanes)
folded = zl.nodes.Zigzag()(compressor, successor=packed)
graph  = compressor.build_static_graph(planar, [folded])

compressor.select_starting_graph(graph)

Two calling conventions show up here and the difference matters. A node object is callable, which builds and registers a graph in one step and returns a GraphID. That form is only available when the node has one outcome to route.

build_static_graph is the general form. It takes a bare NodeID and a list of successors, one per outcome, which is what you need for planar because register_custom_encoder hands back a node and not a graph.

an outcome is not an output transpose_split emits one stream per lane but declares a single variable outcome, so all of its lanes share one successor slot. Asking for two successors is a build error, not a runtime one. Routing lanes apart needs a different tool.

nothing shrinks until the leaf

The useful thing about reading a graph edge by edge is watching the byte count refuse to move. A predictor does not make data smaller, it makes data compressible, and every node before the backend is a rewrite that costs exactly as many bytes as it was given.

Where the 116.5 kB sits is worth a second. The high-level table has planar>zigzag>store_lo landing at 2.00×, and store_lo is this same graph with lane 0 written out raw. Storing 262 144 raw bytes and hitting exactly 2.00× on a 524 288 byte tile means lane 1 rounds to nothing. The high byte of a planar residual over smooth terrain is a sign bit and little else. So the frame you are looking at is the low lane, entropy-coded from 256 kB down to about 116 kB, plus change.

planar the codec openzl does not have

OpenZL ships delta_int, which is a one-dimensional difference and has no idea the stream is a raster. Every geozl predictor exists because that is the wrong prior for a height field, and they enter the graph as custom codecs, a matched encoder and decoder pair that OpenZL calls at the right point in the walk.

The contract is a description both halves must agree on, an id, the input types, and the output types. The encoder writes into an output it asks the state to create and commits an element count. Anything the decoder needs that is not in the stream travels in the codec header, which for planar is the row width, since a flat numeric stream has forgotten it was 512 wide.

the price of a custom codec The id space is yours to version. OpenZL will not check that the encoder that wrote id 101 and the decoder that reads it agree, so you own that compatibility forever. It is also why a compressor holding a custom codec cannot be serialized unless you name the node explicitly.
python · the encoder
PLANAR_ID, WIDTH = 101, 1

class PlanarEncoder(zl.CustomEncoder):
    def multi_input_description(self):
        return zl.MultiInputCodecDescription(
            id=PLANAR_ID,
            name="planar",
            input_types=[zl.Type.Numeric],
            singleton_output_types=[zl.Type.Numeric],
        )

    def encode(self, state):
        src = state.inputs[0].content.as_nparray()
        w   = state.get_local_int_param(WIDTH)
        res = planar_forward(src.reshape(-1, w)).reshape(-1)

        out = state.create_output(0, src.size, 2)
        out.mut_content.as_nparray()[:] = res.view(np.uint16)
        out.commit(src.size)

        state.send_codec_header(np.uint32(w).tobytes())
python · the decoder, registered on the DCtx and not the Compressor
class PlanarDecoder(zl.CustomDecoder):
    def multi_input_description(self):
        return zl.MultiInputCodecDescription(
            id=PLANAR_ID,
            name="planar",
            input_types=[zl.Type.Numeric],
            singleton_output_types=[zl.Type.Numeric],
        )

    def decode(self, state):
        src = state.singleton_inputs[0].content.as_nparray()
        w   = int(np.frombuffer(state.codec_header, dtype=np.uint32)[0])
        rec = planar_inverse(src, src.size // w, w).reshape(-1)

        out = state.create_output(0, src.size, 2)
        out.mut_content.as_nparray()[:] = rec.view(np.uint16)
        out.commit(src.size)

The prediction itself is four lines and all of it happens in uint16. Working in the unsigned view is not a detail you can skip, because the residual has to wrap for the reconstruction to close, and a signed overflow would be undefined at the C level and a numpy warning here.

why zigzag comes next The residual is stored as uint16 but read as signed, and small negatives look like huge unsigned values to an entropy model. zigzag folds them back next to zero, which is the whole reason it sits between every predictor and every backend in the recipe grammar.
python · W + N − NW, with the edges falling back to a plain delta
def planar_forward(a):
    a = a.astype(np.uint16)
    p = np.zeros_like(a)
    p[1:, 1:] = a[1:, :-1] + a[:-1, 1:] - a[:-1, :-1]   # W + N − NW
    p[0, 1:]  = a[0, :-1]                               # first row  → W
    p[1:, 0]  = a[:-1, 0]                               # first col  → N
    return (a - p).astype(np.uint16).view(np.int16)

when static is not enough store_lo

Everything so far was decided before any data arrived. That is a static graph, and it covers most of the recipe grid. It does not cover store_lo, which wants lane 0 sent to store and every other lane sent to entropy, because the two lanes are different kinds of data and deserve different treatment.

The obvious attempt fails, and the error is the clearest statement of the rule.

python · one successor per lane
compressor.build_static_graph(
    zl.nodes.TransposeSplit().base_node,
    [zl.graphs.Store()(compressor), zl.graphs.Entropy()(compressor)],
)
RuntimeError: Graph invalid nb of outcomes (2) is incorrect for node 'zl.transpose_split' (1)

A variable-output node has one outcome no matter how many streams it emits, because the lane count is a property of the data and the graph was built without it. Two lanes for int16, four for int32, and a static shape cannot be right for both.

The tool for this is a function graph, which is a graph whose body is a function that runs during compression with the streams in hand. It receives edges instead of ids, calls run_node to advance them, and set_destination to close each one. Now the lane count is known, because the split already happened.

three ways to branch A static graph is fixed at build time. A selector looks at one stream and picks among graphs you gave it, which is how optim would work if it ran inside OpenZL. A function graph is the general case and can run nodes, inspect their outputs, and route each one.
python · the function graph
class LaneRouter(zl.FunctionGraph):
    def function_graph_description(self):
        return zl.FunctionGraphDescription(
            name="store_lo",
            input_type_masks=[zl.TypeMask.Numeric],
        )

    def graph(self, state):
        edge = state.edges[0]
        (packed,) = edge.run_node(zl.nodes.ConvertNumToStructLE().base_node)
        lanes     = packed.run_node(zl.nodes.TransposeSplit().base_node)

        lanes[0].set_destination(state.custom_graphs[0])   # low byte  → store
        for lane in lanes[1:]:
            lane.set_destination(state.custom_graphs[1])   # the rest  → entropy
python · wire it in, with the two destinations passed as custom graphs
router = compressor.register_function_graph(LaneRouter())
router = compressor.parameterize_graph(router, custom_graphs=[
    zl.graphs.Store()(compressor),
    zl.graphs.Entropy()(compressor),
])

folded = zl.nodes.Zigzag()(compressor, successor=router)
compressor.select_starting_graph(
    compressor.build_static_graph(planar, [folded])
)
planar>zigzag>store_lo 2.00x exact round trip

Passing the destinations as custom_graphs rather than closing over them keeps the router reusable. The same class serves store_lo on a DEM and a different pair of backends on Sentinel-2, where the low byte is sensor noise and the trade lands somewhere else entirely.

compress and decompress CCtx and DCtx

A Compressor is the graph. A CCtx is a run of it, and the two are separate because a graph is expensive to build and cheap to reuse across a whole dataset of tiles.

One parameter is not optional. FormatVersion pins the frame format you emit, and leaving it unset is an error rather than a default, because the right answer is the newest version your readers support and only you know what that is.

what geozl sets for you profile turns the checksum off so the timings are the codec and not the verification. At this level CompressedChecksum and ContentChecksum are both yours, and both default on. Leave them on for anything you intend to keep.
python · compress
cctx = zl.CCtx()
cctx.ref_compressor(compressor)
cctx.set_parameter(zl.CParam.FormatVersion, zl.MAX_FORMAT_VERSION)

frame = cctx.compress([zl.Input(zl.Type.Numeric, tile.reshape(-1))])
524 288 B in 116 522 B out 4.50x
python · decompress, with no compressor in sight
dctx = zl.DCtx()
dctx.register_custom_decoder(PlanarDecoder())

out  = dctx.decompress(frame)
back = out[0].content.as_nparray().view(np.int16).reshape(512, 512)
(512, 512) int16 exact True

The single line that carries the whole design is the one that is missing. DCtx never sees compressor. It reads the frame, finds the codecs the frame says ran, and inverts them, which is what makes every OpenZL frame readable by one decompressor no matter which specialised graph produced it.

The custom codec is the seam in that promise. Drop the register_custom_decoder line and the frame becomes unreadable, not corrupt but unresolvable, because id 101 means nothing to a stock build.

the same seam, upstream This is why the high-level API is worth having. geozl.decompress registers every geozl codec on the context before it touches the frame, so a graph you never chose is still a graph you can read.
python · without the decoder
zl.DCtx().decompress(frame)
RuntimeError: Graph invalid no decoder registered for custom codec id 101

reading a frame and a compressor

FrameInfo parses the header without decompressing anything, which is enough to size a buffer or check that a frame holds what you expected before you pay for the rest of it.

What it will not tell you is the shape or the sign. One numeric output of 524 288 bytes at 2 bytes per element is all the frame knows, and turning that back into a signed 512 by 512 raster is the caller's job, or rumi's.

python
fi = zl.FrameInfo(frame)

fi.num_outputs
fi.output_type(0)
fi.output_content_size(0)
1 Type.Numeric 524288

A compressor can also be written down. serialize gives a compact binary description of the graph and serialize_to_json the readable form, which is the fastest way to check that the thing you built is the thing you meant to build.

The example below is the EOPF baseline, id>transpose>zstd, chosen because it uses nothing but stock codecs. A graph containing planar will refuse to serialize until you give the custom node an explicit stable name, which is the same versioning problem wearing a different hat.

note what is absent No parser, no dtype, no shape, no tile. A serialized compressor is a plan, and it is portable to any machine with the same OpenZL build. It is not a description of your data.
json · compressor.serialize_to_json(), trimmed
{
  "graphs": {
    "#1": {
      "type": "static",
      "node": "zl.transpose_split",
      "successors": [ "zl.zstd#0" ]
    },
    "#2": {
      "type": "static",
      "node": "zl.convert_num_to_struct_le",
      "successors": [ "#1" ]
    },
    "zl.zstd#0": {
      "type": "parameterized",
      "base": "zl.zstd",
      "params": { "ints": { 100: 3 } }
    }
  },
  "start": "#2"
}

The zli command line tool goes further at runtime. zli compress --trace records a trace of the walk that the OpenZL visualizer renders as an interactive graph with the compressed size attributed to every branch, which is the fastest way to find out which lane is actually costing you the frame.

the loop same graph, same bytes

The point of the exercise is that the two paths meet. The recipe string from profile and the graph assembled here are the same object described twice, so the frames compare equal byte for byte.

python
import geozl

rows = geozl.profile(tile)
high = geozl.compress(tile, graph=rows[0]['graph'])   # the recipe string
low  = cctx.compress([zl.Input(zl.Type.Numeric, tile.reshape(-1))])

high == low
planar>zigzag>transpose>entropy 116 522 B == 116 522 B True

Which is the honest argument for staying on the high-level API. Everything on this page is what geozl.compress already does, forty-eight times, before it hands you the winner. Drop down here when you want a predictor that does not exist yet, a routing decision the grid does not offer, or a graph pinned into a pipeline where the search is a cost you refuse to pay twice.

reference

Compressor

build_static_graphnode, listThe general form. Takes a NodeID and one successor GraphID per outcome. Variable outputs count as a single outcome.
node(compressor, successor)→ GraphIDThe shorthand. Node objects are callable, and the successor may be a GraphID or an unparameterized graph object.
parameterize_node→ NodeIDAttaches local_params or a stable name to a node without rebuilding it. Where the tile width reaches planar.
parameterize_graph→ GraphIDThe same for graphs, and how a function graph receives its custom_graphs and custom_nodes.
register_custom_encoder→ NodeIDAdds a codec OpenZL does not ship. The matching decoder goes on the DCtx, not here.
register_function_graph→ GraphIDAdds a graph whose body runs at compression time with the streams available.
register_selector_graph→ GraphIDAdds a graph that inspects one input and returns one of the graphs it was given. Cheaper than a function graph when all you need is a choice.
select_starting_graphGraphIDNames the head. Nothing compresses until this is set.
serialize · serialize_to_json→ bytes · strWrites the graph out. Fails on unnamed custom nodes.

CCtx and DCtx

ref_compressorCompressorBorrows the graph for this context. The compressor stays alive and reusable across every tile in the dataset.
set_parameterCParam, intFormatVersion is required. Also CompressionLevel, DecompressionLevel, CompressedChecksum, ContentChecksum, MinStreamSize, PermissiveCompression.
compresslist[Input] → bytesTakes a list, because a graph may have several inputs. geozl always passes exactly one.
decompressbytes → list[Output]Needs no compressor. Returns one output per leaf the frame declares.
register_custom_decoderCustomDecoderRequired once per custom codec id in the frame, and the seam in the universal decompressor.

Custom codec

multi_input_description→ descriptionThe contract. id, input_types, singleton_output_types and optionally variable_output_types. Encoder and decoder must agree exactly.
state.create_outputidx, elts, widthAllocates output idx with room for elts elements of width bytes. Write through mut_content, then commit the real count.
state.send_codec_headerbytesSide channel for anything the decoder needs that is not in the stream. Read back as state.codec_header.
state.get_local_int_paramkey → intReads a value attached at build time by parameterize_node. Compile-time configuration, not per-frame data.

Function graph

function_graph_description→ descriptioninput_type_masks declares what the graph accepts. custom_graphs and custom_nodes can be declared here or supplied later by parameterize_graph.
state.edgeslist[Edge]One edge per input, each wrapping a live stream you can read with edge.input.
edge.run_nodeNodeID → list[Edge]Runs a codec now and returns one edge per output. This is where a variable-output node finally has a known lane count.
edge.set_destinationGraphIDCloses the branch. Every edge must be either advanced or closed before the function returns.
state.custom_graphslist[GraphID]The graphs handed in at parameterization, in order. Keeps the routing logic independent of the backends it routes to.

Names and signatures track openzl 0.2.0. OpenZL is explicit that its API and codec set will keep moving, so pin the version, and read the upstream docs before porting a graph across a release.
